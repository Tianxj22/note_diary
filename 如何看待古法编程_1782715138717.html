我算是从vscode用到cursor再到Claude Code一路跟过来的用户，对AI编程的看法几经反转。尤其是Claude Code推出之后，我在第一周就开git worktree同时做了4个新feature，完全vibe coding，居然调试都很顺利。我大喜过望，我觉得我可以做产品经理了，技术的事全部交给Claude Code就行了。
问题接踵而至，到了下周当我再开发下一个功能的时候就不灵了，我发现ai写的逻辑并不是最优的，有很多逻辑奇怪且冗余。这个时候下一个功能几乎无法开发，我必须对上一次的代码进行重构。由于AI生成了大量我看不懂的代码，所以我必须努力去理解，这对我的要求很高。我花了不少时间重构完成，项目得以继续。我吃一堑长一智之后，决定每一次ai的提交我都用眼睛去看，这么做之后屎山的问题不严重了，但我在这里意识到了一个矛盾性。就是我开始丢失了编程的感觉，我开始把自己的角色当成了技术经理或者产品经理。而现阶段的AI编程就像一个实习生，不逐行review是不行的，这就要求我非常懂编程！实际上你review别人的代码比自己构思有时候还要难，所以网上也有vibe coding更累了的说法。最终我将这个矛盾性戏谑地总结为：如果我不古法编程，就没办法给出好的prompt，AI coding就会失败；如果我古法编程了，那我为什么还需要AI呢？另一种困难来自于AI coding的等待时间，通常1-2分钟的等待时间不长不短，无论我是去看课还是刷短视频，都会导致我的思维失去连贯性。我很难相信长此以往，我有能力review AI生成的代码并掌控一个复杂项目。说到这，通过对vibe 
coding的爱恨纠缠，逐渐摸索出了和它相处的一点门道。我总结出了其中绝对可以做的事和绝对不能做的事。vibe coding中可以做的事：1. 解决bug绝对有帮助，因为解决bug往往需要先确定root cause，这本质上是一个搜索问题。AI在搜索关键代码方面可谓非常有帮助。而且解决bug是不会怎么产生屎山的，确定问题，人工fix可能也只是几行代码的事。2. 讨论新feature的原型。因为我是全栈，有些功能只是有点灵感，但怎么落地还没头绪。这时候直接开一个新的branch，让AI一路生成一个UI原型，你点一点玩一玩就会有新思路。3. 确定架构。这个部分在没有AI coding的时候我是不太想的，有点凭感觉。有了AI之后我往往会和AI先讨论架构，规定几个关键函数的签名，确定核心数据结构。4. 修改视觉UI，各种flex设计，CSS规范记不住，发现AI对解决视觉问题非常好用。5. 重构，但仅限小型任务。比如我要把这个模块里的某几个方法抽象成一个新的模块，这种迁移AI往往完成得不错。6. 当我完成代码之后，AI对我提意见，Code Review，让AI review我的效果很好，因为我的代码已经写出来了，已经构成一个基础，让它在这个上面提意见，有时真的会发现一些问题。vibe coding中不能做的事：1. 不要完全依赖AI，失去自己对程序的理解。2. 不好好用git commit / branch3. 编程新手不要vibe coding，你把握不住4. 不要在等待代码生成途中刷短视频5. 不要认为编程会被AI取代，就懈怠学习编程技术。恰恰相反，现阶段，只有非常优秀的程序员，才能驾驭这些AI二更（2025.9.19)：感谢大家的喜欢，通过这段时间继续使用的情况，我发现了一些古法编程中不易察觉的优点，在此补充：1. 古法编程使我的大脑更专注古法编程更容易进入做产品的状态，而AI编程则让我的大脑变得异常浮躁，跳脱。大脑和身体一样，不是用的越多越累，其实是用的越多越强大，越有精神。我不相信一个思想上懒惰浮躁的程序员能做出什么好项目（呼应之前谈到的短视频）2. 古法编程带来更好的prompt如果AI编程的结果和prompt的质量呈正相关，那么给出更优秀的prompt的前置条件就是我能够充分地理解代码。因为做一个功能，实现是和AI的讨论一点点生成的，如果我不看代码，那我的prompt就一直模模糊糊，最后的结果也不会好。3. 理解实现有助于新的idea产生因为了解了实现之后，我就非常清楚我的数据库里面表的结构，有什么样子的数据，什么样子的api。那么这些数据和api就像乐高积木，当你灵光一现，把它们重新排列组合一下，可能就有新的功能诞生。5. claude code 2.1.72 + claude opus 4.6 的年代，很多朋友在评论区说 AI 相关的答案都要看一下回答日期，太久之前的回答似乎就像沙丁鱼罐头，时间久了就过期掉。我在把这个半年前写的答案通读了一遍之后，发现里面的大部分内容并未过期，但需要不少修订和补充，特此更新。首先我想说的是一个反直觉的事，就是 AI 在这半年的进步主要不在“写代码”的速度，重点是它正在关注整个软件工程的生态。在半年前，我用 claude code 主要就是写代码，但是现在 claude code 就像一个可控的 openclaw，只要赋予合适的权限，所有我能做的事情它都可以做（token足够的前提下）。接下来，我就分几个关键词展开说一说。（1）Skills （技能）Skills 是这半年最重要的更新之一，它可以理解为一个按需插入的 prompt，同时结合脚本完成一项具体的任务。这个功能使我减少了很多毛刺工作，比如发布 staging 版本，同时发布 5 个平台的客户端，由于 skill 只有一小段 prompt，几乎不会出现幻觉问题，非常可靠。在毛刺工作减少时，我的视野变得更加宽阔，我开始着手做整体的 AI 工作流，也就是输入: idea, 输出: 发布新版本，这其中的所有环节，都有 AI 的参与，这也引出了下一个功能，agent team。(2) Agent team （Agent 团队)我平常有6个项目要维护, 2个服务端，2个客户端，1个小程序，还有1个管理后台。那么心智模型就是1个新的 idea 可能同时需要这6个项目共同参与，协同变更。所以编排整体的 AI 工作流就变得重要了，在这个背景下，我新建了一个 ai-manager项目，每有一个新的idea, 通常我会用 brainstorming skill 开始入手，从 idea 的产生到最终写成 prd.md，是我和claude讨论出来的。有了 prd，通常我会带着 prd 去 lovart 上面设计 UI，这些都完成后，就交各个 teammate 去设计技术实现路径，我会采用官方的 feature dev 命令去完成这个过程。(3) Harness Engineering （驾驭工程）从 Copolit 到 Harness 的这个有趣的比喻也能看出心智模型的变化，之前是我来开，AI 是副驾驶，现在是 AI 来开，人对它约束和管控。人从维护代码的角色变成了维护AI，像 claude code 最新更新的 insights 命令也是基于这个心智模型提出的。这个点着重于强调个性化，也就是让 AI 理解当下的环境，你是如何工作的，怎么命名你舒服，怎么写参数你舒服，怎么做架构你舒服。是测试很多corner case你舒服，还是基础测一下你舒服。这些都没有对错之分，就是像喜欢红色和绿色一样，但是 Harness 就是让 AI 理解人的脾性，进而实现人和 AI 的丝滑合作。(4) Test （测试）测试的重要性来到了它的历史最高点。既然大量的代码都是 AI 通过我的一个简单的 idea 加工生成的，我在把它发布到生产环境的时候，怎么能够确保它是正确的？我觉得答案就是测试。测试分为黑盒和白盒，黑盒测试主要涉及功能测试和线上日志监控，白盒测试则主要是 code review，在我的实践中，测试工程的搭建已经占据我近一半的工作量。很多人提到通过维护 skill 和 claude.md 的方式去让 AI 生成优质的代码，但我认为在严格的测试工程下，劣质的代码一般是无法通过的，即使是黑盒测试。目前测试工程的生态还不是很完善，
这个领域很可能是下半年 Anthropic 发力的点。(5) 真不用看代码了吗 （How dare you!)这部分我的思考是，和半年前有提升，但没有本质区别。在 AI 写好代码后，我仍然需要确认架构图，api 接口的命名，参数类型，dto 的格式等等。以及在关键技术问题的归因上面，我都仍然遇见过 AI 的误判，我险些因为这个误判改变整个功能的技术路线。所以关于这个部分，我仍然坚持之前的看法，我必须是一个优秀的软件工程师，我才能用 AI 写出好的代码。(6) 还需要学习计算机吗 （AI 助教来了）这个想法也没有变，而且我觉得人需要更多地学习。我们对于 AI 的认知很多时候受到网红的影响，比如 Andrej Kapathy 发推说他在去年 11 月，从 80% 手写代码突然变成了 20%；胡渊鸣在新春时节发微信文章，我给 10 个 claude code 打工。最简单的道理，如果学习计算机知识价值在降低，那我和胡渊鸣还有 Andrej 能做出贡献的差距会随着 AI 的发展而缩小，但是事实上这个差距是随着 AI 的发展而拉大的。人必须坚持塑造自己的底层认知，去学半衰期长一些的知识。不过一大利好是在这个时代，学习的毛刺也减少了，我们可以直接吃西瓜的尖尖了！我前些日子想把 CSAPP 捡起来继续学，我直接就和 AI 说，我要学这个了，把网上的资料找一找，做一个 AI 助教出来，结果 AI 直接给我搭出一个保姆级的学习方案，做 labs 的时候简直就像有一个计算机大神在不厌其烦地教我。（可能是因为网上关于这个项目的大神资料太多了）很难想象未来人类的学习速度，不过这个就涉及 AI 教育的领域，不再深入展开了。(7) 还看短视频吗 （真忍不住）我在半年前提出短视频理论之后，大家非常共情。包括我自己，我现在每天和短视频斗智斗勇，有时深刻地觉得现在产出的瓶颈是我自己，我太想摸鱼了，太想看小姐姐了。现在的思路是尽可能把手机放远一点，然后登录电脑微信，和真实的小姐姐聊天（希望对方不是龙虾吧）（手动狗头）。开玩笑啦，其实这个问题的核心是注意力切断，我提出的理念是在 AI 等待时间继续思考这个问题。网文中经常说 openai 的员工开10个agents做事情，我觉得那个应该是 team agents，但是即使开10个，它的思路也是在思考一个问题。人脑是不太适合多线程思考的，我就是这么认为的！注意力不能被切断，切断之后我就会变成一个不能深度思考的人。
时间长了就没能力驾驭 AI 了，这也是之前提到的内容，我坚持并且不改。(8) 最后聊聊AI 在这半年的发展提升了写代码这事的速度，但希望大家从 AI 增益了整个软件工程甚至整个人类脑力生产这个角度去看问题，而不仅仅局限于代码。AI 在做毛刺任务时很有帮助，那么人就要去思考真正重要的事情！很多人失业了吗，我不这么看，我觉得人们本来就不应该做这种xx工作啊。每一个人都有自己独特的经验和有趣的灵魂，尝试走出来，让 AI 帮你实现一个小梦想，从做中学，技术一定会让这个世界更美好！（如果你不知道怎么开始，可以私信我哈，纯社区交流，不是卖课哈哈哈）最后的最后，不要盲目学习网红们的打法，保持克制和理性。我学习打篮球一般也不会看乔丹的录像，大神们的工作流也不一定适合我们，还请理性分析。
